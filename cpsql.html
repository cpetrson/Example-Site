SQL (and noSQL)

Building and Using Database Systems

Spring - Summer 2025

Introduction

I have used a number of databases from cloud environments to local Windows environments to Linux servers. This use has been tied to projects at large organizations as well as setting up small databases for my own projects. An overview:

  • MongoDB (noSQL)
  • MySQL - Windows
  • MySQL - Linux
  • Oracle
  • Others (Courses)

The database chosen depended on the job at hand, whether that was determined by managing levels on large projects, or by the project itself for smaller projects.

Methodology

Each project merited its own specific database and here is a lineup of projects and databases I've utilized:

  1. MongoDB (noSQL)
    • Type: noSQL, cloud-based (cloud.mongodb.com)
    • Purpose: used for a React project
    • Benefits: Cloud access provides a standard environment for client access -- the setup steps can be the same for users from many device types.
    • Notes: Rather than a database with tables, this is a Cluster-Collection model.
  2. MySQL - Windows
    • Type: RDBMS (https://dev.mysql.com/downloads/installer/)
    • Purpose: used for a SQL tutorial, along with SQL Workbench (https://www.Mysql.com/products/workbench/)
    • Benefits: Smaller database that provided the backend for a JavaScript app
  3. MySQL - Linux
    • Type: RDBMS
    • Purpose: used for creating a LAMP Stack Server with WordPress
    • Benefits: A very common stack for serving up websites, the LAMP Stack consists of Linux, Apache, MySQL, and PHP. This is a free and open source stack.
    • Notes: As the 'M' in the LAMP Stack, MySQL has played a key role in powering websites from tiny to enterprise.
  4. Oracle
    • Type: RDBMS
    • Purpose: Enterprise-level database management system
    • Benefits: Robust 'flagship' product
  5. Others
    • Type: RDBMS
    • Purpose: Structured Courses
    • Benefits: Practice with RDBMS tables, data, and reporting

Findings

Some basic similarities between databases:
  1. SQL is very similar between databases - standard operations as well as aggregation functions and JOINS are very similar.
    • SELECT
    • INSERT
    • DELETE
    • UPDATE
    • GROUP BY
    • COUNT()
    • AVG()
    • SUM()
    • JOINS (LEFT, INNER, RIGHT and the less-used OUTER)
  2. The standard tasks in a database lifecycle are still present - however varied the project, these are steps that remain integral to database management:
    • Database creation and Schema
    • User creation and permissions
    • Accessing the database with a GUI or visualizer
    • Adding test data to ensure that things are working
    • Accessing the database programmatically
    • Finetuning indexes
    • Searching
    • Adding data
    • Viewing, reporting and real-time updating
Some differences between databases:

The first main difference between databases that I've experienced has been in the application layer when connecting to the database. Based on the packages used to connect to the database, the syntax looks different but the underlying tasks of communicating with the database remain the same.

The second major difference has been the shift from relational databases to the NoSQL-type databases with a more unstructured format.

Conclusion

Databases are the warehouses of the Internet. Not only do they power the dynamic user experiences we have come to expect: everything from logging into our favorite sites and having our own accounts, to 'following' our favorite content providers, and the very purchases we make online - but they also undergird the newer push toward 'suggested' content and advertiser-identifiers. It all runs on databases.
